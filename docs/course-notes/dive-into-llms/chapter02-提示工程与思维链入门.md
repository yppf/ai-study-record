# Chapter 2 知识总结：提示工程与思维链入门

对应 PDF：`dive-into-prompting.pdf`

这章主要讲的是：

`如果不急着训练模型，而是先想办法把问题问得更清楚、更聪明，能不能让大模型表现明显变好？`

答案是：`能，这就是提示工程和思维链技术的核心出发点。`

---

## 1. 这章到底在讲什么

这章其实在回答两个很重要的问题：

1. `模型怎么用，效果会更好？`
2. `面对复杂问题，怎么引导模型一步一步推理？`

所以它的主线可以理解成：

- 先学会怎么提问
- 再学会怎么让模型按步骤想
- 最后再扩展到更复杂的推理结构和 Agent 应用

这也是为什么 chapter2 会比 chapter1 更偏“使用方法”和“推理方式”。

---

## 2. 什么是提示工程

提示工程（Prompt Engineering）就是：

`为了让大模型更好完成任务，系统地设计和优化提示词。`

最白的话：

`研究怎么提问，才能让模型更好地干活。`

提示工程一般会做这些事：

- 把任务说清楚
- 给足上下文
- 规定输出格式
- 必要时给示例
- 复杂任务时拆步骤

所以提示工程不是训练模型，而是：

`更聪明地使用模型。`

---

## 3. 常见提示方式怎么理解

### 3.1 Zero-shot

不给示例，直接让模型做。

可以理解成：

`不演示，直接出题。`

### 3.2 One-shot / Few-shot

先给 1 个或几个例子，再让模型照着做。

可以理解成：

`先做一道例题，再让模型做新题。`

### 3.3 In-context learning

模型不重新训练，只靠当前输入中的上下文和示例，就在当前会话里“学会”怎么做任务。

可以理解成：

`现学现用。`

---

## 4. 什么是思维链（CoT）

思维链（Chain-of-Thought, CoT）就是：

`不要让模型直接报答案，而是让它先把推理过程写出来。`

最白的话：

`让模型一步一步想。`

这在数学题、逻辑题、常识推理题里特别常见。

例如经典提示：

`Let's think step by step.`

---

## 5. 为什么 CoT 有用

因为很多时候模型答错，不是因为它完全不会，而是因为：

- 跳步了
- 漏条件了
- 推理链断了

CoT 的作用是：

- 把问题拆开
- 让中间过程显式化
- 降低“拍脑袋给答案”的概率

所以 CoT 的核心不是让模型“更啰嗦”，而是：

`让推理过程更完整、更可见。`

---

## 6. 从普通 CoT 到更复杂结构

这一章后面其实已经不只是讲最基础的 CoT，而是在讲：

`如果一条线的思路不够，推理过程还能怎么组织？`

典型扩展包括：

### 6.1 Self-Consistency

多次生成不同推理链，再投票选答案。

最白的话：

`多想几次，看多数结果。`

### 6.2 Program of Thought（PoT）

把可计算部分交给程序。

最白的话：

`想法模型来给，计算程序来做。`

### 6.3 Tab-CoT

把推理过程写成表格。

最白的话：

`把步骤列成表，更整齐。`

### 6.4 Tree-of-Thought（ToT）

多分支探索，而不是一条链走到底。

最白的话：

`像走迷宫一样，边试边选。`

### 6.5 Graph-of-Thought（GoT）

让不同思路像图一样交叉、汇合、回溯。

最白的话：

`把推理变成一张网络。`

---

## 7. 提示工程不只是“写提示词”

这章最重要的升级理解之一是：

`现代提示工程越来越像任务流程设计。`

因为你后面会看到很多方法已经不只是：

- 给一句提示词

而是：

- 先查资料
- 再思考
- 再调工具
- 再反思
- 再输出

所以提示工程在今天更像：

`如何为模型设计一套完成任务的工作流程。`

---

## 8. RAG、ReAct、Reflexion 和 Agent 是怎么接进来的

chapter2 后半段其实已经把你带到更实战的地方了。

### 8.1 RAG

RAG = `先查资料，再回答`

适合：

- 知识库问答
- 最新信息
- 私有文档

### 8.2 ReAct

ReAct = `边想边做`

让模型：

- 推理
- 调工具
- 看结果
- 再继续

### 8.3 Reflexion

Reflexion = `做完再改`

让模型：

- 回顾哪里错了
- 总结经验
- 下一轮修正

### 8.4 Agent

当这些能力组合起来时，模型更像一个：

`能查、能做、能改的任务执行者`

- `Prompt engineering is not only about wording, but also about structuring the task clearly.`
  - 提示工程不只是措辞问题，更是把任务结构讲清楚。

- `Chain-of-thought prompting helps the model expose its intermediate reasoning steps.`
  - 思维链提示能帮助模型把中间推理步骤显式写出来。

## 11. 难点与易混淆

### 11.1 提示工程 vs 提示学习

| 概念 | 更偏什么 |
|---|---|
| 提示工程 | 使用方法，怎么把提示写好 |
| 提示学习 | 研究概念，用 prompt 让模型适应任务 |

### 11.2 CoT vs ToT vs ReAct vs Reflexion

| 方法 | 最白的话 |
|---|---|
| CoT | 一步一步想 |
| ToT | 多分支探索 |
| ReAct | 边想边做 |
| Reflexion | 做完再改 |

### 11.3 RAG vs 微调

| 方法 | 更像什么 |
|---|---|
| RAG | 回答前先查资料 |
| 微调 | 把知识或能力训练进模型 |

---

## 12. 延伸补充：为什么这章对转行者特别重要

### 12.1 因为它最贴近“怎么把模型用起来”

很多转行者一开始最容易焦虑的是：

`我不可能一上来就训练基础模型，那我到底能做什么？`

这章给的答案非常现实：

- 你可以先学会写提示
- 再学会 few-shot
- 再学会 CoT
- 再学会 RAG
- 再学会简单 Agent 工作流

也就是说：

`不用从造模型开始，也能做出很有价值的东西。`

### 12.2 因为今天很多实际应用首先靠的不是“重训练”，而是“更好地用”

现实里很多项目一开始不会先上复杂微调，而是先做：

- 提示工程
- RAG
- 工具调用
- 工作流编排

这也是转行者最容易短时间上手的方向。

### 12.3 为什么后面越学越像“系统设计”

一开始你会觉得提示工程就是写 prompt。  
但学到后面你会发现，它越来越像：

- 任务拆解
- 推理设计
- 工具协作
- 错误恢复

这其实已经接近：

`小型智能系统设计。`

---

## 13. 本章只记 3 句

- chapter2 的核心是：不用先训练模型，也可以通过更好的提示和流程设计大幅提升效果。
- CoT 是让模型一步一步想，而后续的 ToT、ReAct、Reflexion 是对推理结构和执行流程的进一步扩展。
- 对转行者来说，提示工程、RAG 和简单 Agent 是非常值得优先掌握的实用能力。

---

## 14. 最白话总结

`这章讲的是：大模型不只是“问一句答一句”，如果你会更清楚地提问、会给示例、会让它一步一步想、会让它先查资料再行动，它就会表现得聪明很多。这就是提示工程和思维链技术的价值。`
