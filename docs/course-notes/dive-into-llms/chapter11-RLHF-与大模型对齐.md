# Chapter 11 知识总结：RLHF 与大模型对齐

对应 PDF：`RLHF.pdf`

这章主要讲的是：

`怎样让大模型不只是会生成语言，还更符合人的偏好、更像一个“愿意配合、相对安全、输出更讨喜”的助手。`

---

## 1. 这章到底在讲什么

预训练后的大模型很强，但有个问题：

`它未必天然符合人的意图。`

比如它可能：

- 会答，但答得很奇怪
- 会生成，但不够有帮助
- 会配合，但不够安全
- 在某些微小训练变化后，行为就变坏

所以“对齐”要解决的问题是：

`怎么让模型的行为更稳定地符合人类希望的方向。`

---

## 2. 什么是对齐

对齐（Alignment）可以简单理解成：

`让模型的目标更贴近人的目标。`

这件事不是只靠“教知识”就能解决，因为很多时候问题不是：

- 模型知不知道

而是：

- 模型会不会按人希望的方式回答

例如：

- 更有帮助
- 更礼貌
- 更安全
- 更少胡说

---

## 3. 为什么 RLHF 很关键

RLHF 的全称是：

`Reinforcement Learning from Human Feedback`

中文就是：

`基于人类反馈的强化学习`

最白的话解释：

`让人来告诉模型，什么回答更好，然后再据此优化模型。`

它之所以重要，是因为：

`很多“好回答”并没有唯一标准答案，但人能比较哪个更好。`

---

## 4. RLHF 的核心流程怎么理解

你可以把 RLHF 流程理解成三步：

### 4.1 模型先生成回答

面对一个问题，模型可以生成不同候选回答。

### 4.2 人类做偏好比较

人类不一定写出标准答案，但可以判断：

- A 比 B 更好
- B 比 A 更清楚
- 哪个更安全
- 哪个更有帮助

### 4.3 用这些偏好继续优化模型

模型逐渐学会：

`哪些输出更容易得到“好反馈”。`

所以 RLHF 的核心不是“教知识”，而是：

`教模型更符合人类偏好地行动。`

---

## 5. 为什么不是只做监督微调就够了

监督微调（SFT）当然很重要，但它通常更偏向：

- 给任务样例
- 学会按格式做
- 学会回答某类问题

而 RLHF 更进一步关注：

- 哪种回答更受人喜欢
- 哪种行为更安全
- 哪种表达更像一个好助手

所以可以简单理解成：

- SFT：`教模型会做事`
- RLHF：`教模型把事做得更像人想要的样子`

---

## 6. 这章为什么会提到 PPO

PDF 里专门讲了：

- 从强化学习到人类反馈学习
- 从策略梯度到 PPO

这说明在 RLHF 里，PPO 是一个非常经典的优化方法。

你不需要一开始就学透数学公式，但要知道 PPO 在这里扮演什么角色：

`它负责根据奖励信号，稳定地优化语言模型的策略。`

最白的话：

`模型先回答，再根据奖励慢慢往更优方向调。`

---

## 7. 为什么 RLHF 要有奖励模型或奖励信号

如果没有奖励，模型不知道：

- 哪种输出更好
- 哪种输出更差

而人类偏好本身很难直接变成训练信号，所以常见做法是：

- 用人工比较数据训练一个奖励模型
- 或在示例实验里直接用某个外部模型分数当奖励

这章 notebook 里的例子，就是用情感分类器给 GPT-2 的输出打分。

也就是说：

`“奖励”是把“人类偏好”变成可优化信号的关键桥梁。`

---

## 8. 这章里的实践例子到底在做什么

README 和 notebook 给了一个非常典型的教学案例：

`让 GPT-2 学会生成更积极的 IMDB 电影评论。`

流程是：

1. 准备 IMDB 文本
2. 让 GPT-2 生成后续评论
3. 用一个情感分类器打分
4. 把“更积极”的分数作为奖励
5. 用 PPO 优化 GPT-2

这不是在追求最真实的工业级 RLHF，而是在用一个最容易理解的例子说明：

`如何把外部评价信号变成模型优化方向。`

---

## 9. KL 约束为什么重要

在 RLHF/PPO 里，还有一个很关键的概念：

`不要让优化后的模型偏离原模型太远。`

这是因为如果只为了奖励拼命优化，模型可能会：

- 输出变得很怪
- 过度投机
- 为了拿高分失去自然性

所以常会加一个 KL 约束，让模型别离原本的语言分布太远。

最白的话：

`既要更符合偏好，也别优化得太离谱。`

---

## 10. RLHF 的价值到底是什么

### 10.1 让模型更像助手

不仅会答，还更会“配合人”。

### 10.2 让模型输出更符合偏好

比如：

- 更积极
- 更礼貌
- 更有帮助

### 10.3 让模型学会一些靠标准答案难以表达的东西

很多任务没有唯一正确答案，但人能比较“哪个更好”。

这正是 RLHF 擅长的地方。

---

## 11. RLHF 的局限也要知道

### 11.1 奖励可能被投机利用

模型可能学会“骗奖励”，而不是真的更好。

### 11.2 人类偏好本身不总一致

不同人眼中的“好回答”可能不同。

### 11.3 对齐可能很脆弱

PDF 开头提到的“轻微训练就可能导致行为崩坏”，就是在提醒：

`对齐不是一劳永逸的。`

### 11.4 成本高

需要：

- 人类偏好数据
- 奖励建模
- 强化学习优化

---

## 12. RLHF 和安全是什么关系

RLHF 不是只为了“让回答更好看”，它也和安全密切相关。

因为很多安全目标本质上也是偏好目标：

- 不要输出危险内容
- 更谨慎
- 不要乱答
- 不要明显违背人类意图

所以 RLHF 往往是大模型安全对齐的重要组成部分。

但它也不是安全的全部，因为：

`对齐好了，也不代表绝对不会被越狱或不会失控。`

---

## 13. 对转行学习者最值得掌握的理解

你不需要一开始就把 PPO 数学推导全啃明白。  
先抓住下面这条主线就很够用了：

`预训练让模型会说话，监督微调让模型会做任务，RLHF 让模型更像人愿意使用的助手。`

这句话在你以后理解 ChatGPT、Claude、Gemini 这些系统时会非常有用。

---

## 14. 这一章最值得记住的 3 句话

- RLHF 的核心是：用人类偏好或其近似信号来优化模型行为。
- 它解决的不是“模型知不知道”，而是“模型会不会以人希望的方式去回答和行动”。
- PPO 等方法在 RLHF 里扮演的是“如何根据奖励稳定优化模型”的角色。

---

## 15. 最白话总结

`这章讲的是：如果说预训练让模型学会了语言，监督微调让模型学会了任务，那 RLHF 就是在教模型“什么样的回答更像一个让人愿意继续使用的助手”。`

## 17. 难点与易混淆

### 17.1 SFT vs RLHF

| 方法 | 更像在解决什么问题 |
|---|---|
| SFT | 会不会按要求做事 |
| RLHF | 做出来的方式是不是更符合人类偏好 |

### 17.2 奖励高不代表真的更“好”

这是 RLHF 最容易忽视的点。

模型可能学会：

- 拿高分
- 讨好奖励模型

但不一定真的变得更可靠。  
所以 RLHF 很强，但也必须警惕“奖励投机”。

---

## 18. 延伸补充：RLHF 在今天的大模型训练流程里通常处在什么位置

通常可以把完整流程理解成：

1. 预训练
2. 指令微调 / SFT
3. 奖励建模
4. RLHF / PPO 优化

也就是说，RLHF 一般不是从零开始训练模型，而是建立在：

- 一个已经会语言的模型
- 一个已经基本会按要求做任务的模型

之上。

### 18.1 为什么先 SFT 再 RLHF

因为如果模型连基本任务都不会做，直接上 RLHF 很难训练。

SFT 的作用更像：

`先把模型扶到“会答题、会配合”的水平`

然后 RLHF 再进一步把它往：

- 更符合偏好
- 更自然
- 更安全

的方向推。

### 18.2 RLHF 为什么经常和“聊天助手”强相关

因为聊天助手这种产品最典型的特点就是：

- 没有唯一标准答案
- 用户对“回答质量”的感受差异很大
- 输出风格、帮助程度、安全性都很重要

这恰好就是 RLHF 最擅长发挥作用的地方。

### 18.3 对齐为什么仍然脆弱

即使做了 RLHF，也不代表模型永远安全或稳定。  
因为：

- 奖励不可能覆盖所有情况
- 偏好本身会变化
- 模型还可能被越狱

所以 RLHF 很重要，但不该被理解成“安全终点”，更像：

`对齐体系中的关键一环。`

## 19. 补充：把强化学习基本概念一一对应到大模型里

课件里讲的强化学习基本概念，放到大模型训练里可以这样理解：

- `状态（state）`：prompt + 当前已经生成的前缀
- `动作（action）`：生成下一个 token，或者粗略看成生成整段回答
- `策略（policy）`：当前语言模型在不同上下文下输出 token 概率分布的方式
- `奖励（reward）`：奖励模型、人工偏好，或可验证任务给出的分数
- `价值函数（value function）`：从当前生成状态继续往后走，最终大概还能拿到多少总奖励
- `环境（environment）`：在大模型里通常不是物理世界，而更像任务输入、数据集、上下文分布

最值得记住的一点是：

`大模型里的 RL 和教材里“机器人在世界中行动”的 RL 原理相通，但形式更抽象。`

---

## 20. 为什么大模型里的强化学习和传统 RL 不完全一样

传统 RL 常常是：

`状态 -> 动作 -> 环境变化 -> 新状态 + 奖励`

但在很多 LLM 场景里，更像是：

`给一个 prompt -> 模型生成回答 -> 奖励模型或评测器打分`

所以它有几个特点：

- 环境更弱，很多时候主要是数据分布而不是真实动态世界
- 任务常常接近“单轮 episode”，奖励往往在整段回答后统一给出
- 奖励很多时候不是天然存在，而是人为构造出来的

所以更准确地说：

`RLHF 是把文本生成过程形式化成序列决策问题，再用强化学习方法优化。`

---

## 21. 从策略梯度到 PPO，该怎么抓主线

### 21.1 Value-Based、Policy-Based、Actor-Critic

- Value-Based：先学动作值，再选最大值动作，典型是 Q-Learning / DQN
- Policy-Based：直接学动作概率分布，典型是 REINFORCE / Policy Gradient
- Actor-Critic：一边学策略，一边学价值评估，典型是 PPO

对大模型来说，更自然的是 Policy-Based 或 Actor-Critic，因为语言模型本来就在输出下一个 token 的概率分布。

### 21.2 最原始的 Policy Gradient 在做什么

最直白的理解就是：

`哪条生成轨迹最后奖励更高，就提高这条轨迹中动作再次出现的概率。`

REINFORCE 的问题在于，它常常把整条回答的总奖励比较粗糙地分配给轨迹中的很多 token，因此：

- 方差大
- 训练容易抖
- 信用分配不够精细

### 21.3 PPO 比原始 PG 多了什么

PPO 可以简单理解成：

`更稳定、更保守的策略梯度。`

它通常结合了：

- Actor：语言模型本身，负责生成动作
- Critic：价值估计器，负责评估“这一步到底比平均水平好多少”
- 约束更新幅度的机制：防止策略一次改太猛

所以 PPO 的核心思想不是“更激进地追求高奖励”，而是：

`在 advantage 的指导下，小步、受约束地把模型往高奖励方向推。`

---

## 22. 为什么说“动作本身不可导”

这不是“逆推”的意思，而是说：

`模型最终做的是离散选择，而离散采样/argmax 这一步不能像连续值那样直接传梯度。`

例如模型给出：

- 猫：0.7
- 狗：0.2
- 鸟：0.1

概率分布本身是可导的，但“最后抽中了猫”这件事是离散事件，不是平滑连续函数。  
因此，如果整段回答最后才得到一个奖励分数，就不能像普通监督学习那样，直接顺着“选中了哪个 token”这一步做标准反向传播。

所以强化学习里要用策略梯度、PPO 这类方法，去间接估计：

`哪些动作会带来更高奖励，就提高它们的概率。`

---

## 23. PPO 和 DPO 的区别

### 23.1 PPO

PPO 是经典 RLHF 路线，通常包含：

1. 先训练奖励模型
2. 让模型采样回答
3. 奖励模型给分
4. 用 PPO 更新策略

特点是：

- 需要 reward model
- 常常需要 critic / value head
- 是显式强化学习
- 更通用，但训练更复杂

### 23.2 DPO

DPO（Direct Preference Optimization）更像是：

`直接用偏好对 chosen / rejected 来训练模型，不显式走 PPO 那套 RL 流程。`

它通常不单独训练 reward model，也不需要显式 actor-critic 结构，工程实现更简单。

### 23.3 怎么理解两者关系

- PPO：更像“先打分，再做强化学习”
- DPO：更像“直接利用偏好对做可微优化”

简单偏好对齐任务里，DPO 往往更省事；  
更一般、奖励更复杂、需要在线探索或可验证奖励的任务里，PPO 和其他 RL 方法仍然很重要。

---

## 24. 训练阶段和推理阶段不要混

RLHF 那张图常把很多模块画在同一个闭环里，但最好拆成两个阶段理解：

### 24.1 阶段一：先训练奖励模型

- 让模型生成多个回答
- 人类比较哪个更好
- 用这些偏好数据训练 Reward Model

### 24.2 阶段二：再用奖励模型训练语言模型

- 给一个 prompt
- 模型生成回答
- Reward Model 打分
- PPO 等算法更新模型

### 24.3 真正上线推理时

通常只有：

- 用户输入 prompt
- 模型直接回答

也就是说，Reward Model 多数是训练时使用，不是每次用户提问时都在后台实时参与。

## 25. 安全对齐补充：显式安全信号与安全层

除了 RLHF 这种“总体偏好优化”，课件还提到了更直接的安全思路。

### 25.1 显式安全信号

一个重要想法是：

`把“识别恶意请求”显式做成一个二分类任务。`

也就是训练模型先判断输入是：

- 良性
- 恶意 / 危险

这样安全能力就不只是隐式藏在生成模型里，而是变成一个明确监督目标。

### 25.2 安全层

课件还讨论了模型内部是否存在“安全层”，也就是：

`模型在某些中间层开始明显区分正常请求和恶意请求。`

常见验证方法是：

- 对比正常输入和恶意输入在不同层的表征
- 看余弦相似度是否突然下降
- 看角度差异是否突然上升

如果差异在某些层显著放大，就说明模型可能在这些层开始进行“安全判断”。

### 25.3 这和 RLHF 的关系

- RLHF：更像从整体偏好上把模型往更安全、更有帮助的方向推
- 显式安全分类：更像给模型加一个明确的“危险识别任务”

两者不是互相替代，而是可以配合使用。

## 26. 这一章新增最值得记住的 6 句话

- 大模型里的策略，就是模型在上下文中生成下一个 token 的概率分布。
- 大模型里的环境通常比传统 RL 更弱，很多时候更像数据集和任务分布。
- PPO 本质上是“更稳定、更受约束的策略梯度”。
- “动作不可导”不是不能训练，而是离散采样这一步不能像连续变量那样直接传梯度。
- DPO 是直接用偏好对优化模型，PPO 是先建奖励再做强化学习。
- 安全不一定只体现在最终拒答上，模型内部也可能存在专门做安全判断的层。
